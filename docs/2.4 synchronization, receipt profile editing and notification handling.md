# 2.4 Synchronization, Receipt Profile Editing and Notification Handling

This document summarizes synchronization improvements, receipt handling, and notification system enhancements implemented in TavernAI, with main file paths, implementation strategies, and key decisions.

## 1) Improved synchronization in waiter station (multi-device support)

**What it does**
- Ensures correct table association when marking items as done across multiple waiter devices
- Prevents cross-table item resolution bugs when the same item_id exists in different table sessions
- Properly scopes WebSocket connections per station to prevent event leakage between roles

**Key files**
- backend/app/storage/sqlite.py (table-scoped item operations)
- backend/app/storage/sqlalchemy_adapter.py (TableSession ownership validation)
- backend/app/storage/base.py (Storage interface return type updates)
- backend/app/main.py (receipt_id validation and WebSocket station filtering)
- mobile-app/src/pages/HomePage.jsx (WebSocket cleanup on logout/tab switch)
- mobile-app/src/components/WaiterView.jsx (WebSocket lifecycle management)
- mobile-app/src/components/StationView.jsx (WebSocket cleanup)

**Strategy and decisions**
- Storage backends now enforce table ownership for all item operations (update_order_status, delete_order, get_order_by_id)
- SQLiteStorage adds `(OrderModel.table_id == table_id)` conditions to WHERE clauses
- SQLAlchemyStorage validates TableSession ownership before allowing item modifications
- `delete_table()` method signature updated to return `Optional[int]` for receipt_id (None for non-receipt-supporting backends)
- Main.py validates receipt_id before broadcasting `receipt_ready` events to prevent sending invalid IDs
- WebSocket connections are properly closed during logout via forced component unmount
- Added unique React keys to WaiterView and StationView to ensure complete unmount on tab/role switches
- Logout flow now waits for component cleanup (100ms delay) before navigation to ensure WebSocket connections close

**Before/After Comparison:**
- **Before:** `get_order_by_id(table_id, item_id)` ignored table_id parameter, causing item resolution across wrong tables
- **After:** All item lookups strictly scoped to `(table_id, item_id)` pair with ownership validation
- **Before:** Logging out and logging back in with different role left old WebSocket connections active
- **After:** Logout forces component unmount and WebSocket cleanup before role/tab switch


## 2) Receipt profile editing and generation

**What it does**
- Generates receipts with restaurant profile information (name, phone, address)
- Includes restaurant metadata in receipt content for printing
- Splits receipt-related WebSocket events to prevent workstation interference

**Key files**
- backend/app/storage/sqlalchemy_adapter.py (receipt generation with RestaurantProfile)
- backend/app/api/receipts_router.py (receipt endpoints and content structure)
- backend/app/db/models.py (RestaurantProfile, Receipt, TableSession)
- backend/app/main.py (WebSocket event splitting: table_finalized vs receipt_ready)
- mobile-app/src/pages/ReceiptView.jsx (receipt display with restaurant info)
- mobile-app/src/components/WaiterView.jsx (receipt_ready event handler)

**Strategy and decisions**
- Receipt content includes restaurant profile data embedded as JSON
- Restaurant profile is fetched from database during table finalization (delete_table)
- Receipt record links to primary Order via order_id foreign key
- WebSocket events split into two separate actions:
  - `table_finalized`: Broadcast to ALL stations (cleanup only, no receipt_id)
  - `receipt_ready`: Sent ONLY to "waiter" station (includes receipt_id for opening print tab)
- This separation prevents receipt tabs from opening on workstation displays (kitchen, grill, drinks)
- Receipt ID is integer from database auto-increment, not string concatenation
- Empty receipt handling: returns None when no orders exist for a session

**Receipt content structure:**
```json
{
  "restaurant": {
    "name": "My Taverna",
    "phone": "+30 123 456 789",
    "address": "123 Main Street"
  },
  "table": "4",
  "created_at": "2026-03-03T14:30:00+02:00",
  "closed_at": "2026-03-03T16:45:00+02:00",
  "items": [
    {
      "name": "Greek Salad",
      "quantity": 2,
      "unit_price": 8.50,
      "line_total": 17.00,
      "status": "done"
    }
  ],
  "total": 42.50
}
```


## 3) Waiter notifications when items are ready

**What it does**
- Sends real-time notifications to waiter devices when kitchen/grill/drinks marks items as done
- Includes table number and item details in notification message
- Prevents duplicate notifications through deduplication map with time-based expiry
- Supports both visual toast notifications and optional sound alerts

**Key files**
- backend/app/main.py (HTTP and WebSocket mark_done handlers with notification broadcast)
- mobile-app/src/components/WaiterView.jsx (notification handling and display)
- mobile-app/src/store/notificationStore.js (notification state management)
- mobile-app/src/utils/sounds.js (notification sound playback)

**Strategy and decisions**
- Notifications sent via WebSocket action `"notify"` with message payload
- HTTP endpoint `/item/{item_id}/done` broadcasts notification after marking item as done
- WebSocket `mark_done` action also triggers notification to waiter station
- Notification message format: Greek text with table number (e.g., "ετοιμα Greek Salad τραπέζι 4")
- Deduplication prevents same notification showing multiple times using Map with TTL (15 seconds default)
- Notifications auto-dismiss after TTL expires (configurable per notification)
- Max 6 notifications shown simultaneously (newest first)
- Sound alerts play on notification arrival (can be muted via icon button)
- Notification guard timeout prevents duplicate receipt/notification handling

**Notification flow:**
1. Station (kitchen/grill/drinks) marks item as done via WebSocket or HTTP
2. Backend updates item status to "done" in storage
3. Backend broadcasts `{"action": "notify", "message": "...", "table": N}` to "waiter" station
4. Waiter device receives notification via WebSocket
5. WaiterView displays toast notification with auto-dismiss
6. Optional sound alert plays (if not muted)


## 4) Debug log cleanup

**What it does**
- Removed verbose debug logging added during troubleshooting
- Kept essential error logs and warnings for production monitoring
- Cleaned up WebSocket connection lifecycle logs
- Removed receipt_ready broadcast debugging

**Key files**
- backend/app/main.py (WebSocket and broadcast logging)
- mobile-app/src/services/api.js (HTTP request/response logging)
- mobile-app/src/components/WaiterView.jsx (notification and WebSocket logs)
- mobile-app/src/components/StationView.jsx (WebSocket message logs)
- mobile-app/src/pages/HomePage.jsx (logout flow logs)

**Logs removed:**
- `[broadcast_to_station] ALL ACTIVE STATIONS: ...` (receipt debugging)
- `[broadcast_to_station] Sending receipt_ready to station 'X'` (receipt debugging)
- `[WebSocket] Station 'X' connected. Total connections: N` (connection tracking)
- `[WebSocket] Station 'X' disconnected. Remaining: N` (connection tracking)
- `[StationView] Mounting WebSocket for station "X"` (component lifecycle)
- `[StationView] Unmounting WebSocket for station "X"` (component lifecycle)
- `[StationView.drinks] Received message: new` (message tracking)
- `[HomePage] Logout initiated - forcing component unmount` (logout flow)
- `[HomePage] Components unmounted, proceeding with logout` (logout flow)
- `[getConfig] called, cache exists?` (API calls)
- `[buildHttpUrl] getting config for path` (API calls)
- `[api] POST response status: 200` (API responses)

**Strategy and decisions**
- Retained error-level logs for exception tracking and debugging production issues
- Kept user-facing console logs for critical events (e.g., WebSocket connection failures)
- Production logging focuses on errors and warnings, not info/debug messages
- Environment variable gating still available for access logs (`QUIET_ACCESS_LOGS=1`)
- Removed notification-specific debug instrumentation (`NOTIFICATION_DEBUG` environment variable)
